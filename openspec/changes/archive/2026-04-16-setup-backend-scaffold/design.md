## Контекст

Проект стартовал без документированной истории принятия решений по первоначальной настройке инфраструктуры бэкенда. Данное изменение (Change) ретроспективно фиксирует решения, принятые при создании каркаса Spring Boot, на основе которого строятся все последующие функции.

Текущее состояние:
- `backend/pom.xml` — Maven сборка с родительским артефактом Spring Boot 3.4.4
- `backend/src/main/java/com/knowledgebase/` — корневой пакет приложения
- `backend/src/main/resources/` — файлы конфигурации и шаблоны
- PostgreSQL для production-окружения, H2 для тестов

## Цели / Не цели

**Цели:**
- Создать воспроизводимую структуру проекта для бэкенда на Spring Boot
- Задокументировать ключевые инфраструктурные решения для использования в будущем
- Обеспечить корректное версионирование всех зависимостей

**Не цели:**
- Отсутствие бизнес-логики (аутентификация, CRUD и т.д.)
- Отсутствие конфигурации развертывания (Deployment/CI/CD)
- Изменения на фронтенде

## Принятые решения

### 1. Spring Boot 3.x вместо 2.x
**Решение:** Использовать Spring Boot 3.4.4 (последняя версия 3.x на момент настройки).
**Обоснование:** Базовый стандарт Java 17, переход на пространство имен Jakarta EE, долгосрочная поддержка. Версия 2.x достигла конца жизненного цикла.

### 2. Структура пакетов на основе Clean Architecture
**Решение:** Организовать код по слоям, а не по функциям:
```
com.knowledgebase.interfaces.rest       — Контроллеры, DTO
com.knowledgebase.domain                — Сущности, доменные сервисы
com.knowledgebase.infrastructure        — Репозитории, внешние интеграции
com.knowledgebase.application           — Сценарии использования / сервисный слой
com.knowledgebase.config                — Классы конфигурации Spring
```
**Обоснование:** Четкое разделение ответственности, простота навигации по коду в зависимости от задачи.

### 3. Maven вместо Gradle
**Решение:** Использовать Maven с файлом `pom.xml`.
**Обоснование:** Более прост в освоении для команды, незнакомой с Gradle, лучшая интеграция со средами разработки для Java-команд.

### 4. Liquibase вместо Flyway
**Решение:** Использовать Liquibase для управления миграциями базы данных.
**Обоснование:** Поддержка форматов изменений XML/YAML/SQL, встроенная поддержка отката (rollback), независимость от конкретной СУБД.

### 5. Thymeleaf для серверного рендеринга
**Решение:** Добавить зависимость `spring-boot-starter-thymeleaf` наряду с REST контроллерами.
**Обоснование:** Некоторые страницы (например, административные панели) выгоднее рендерить на сервере без необходимости поднимать отдельный фронтенд.

### 6. JGit для доступа к Git-репозиториям
**Решение:** Включить библиотеку `org.eclipse.jgit` для программной работы с Git.
**Обоснование:** База знаний хранит документы непосредственно в Git-репозиториях.

## Риски и компромиссы

- [Увеличение поверхности атаки из-за Thymeleaf] → Использовать только для административных страниц, сохранив REST API в качестве основного интерфейса.
- [Сложности компиляции из-за Lombok] → Смягчается явным указанием порядка обработчиков аннотаций (annotation processor) в `pom.xml`.
- [Различия в диалектах H2 и PostgreSQL] → Используется единый JPA/Hibernate, но интеграционные тесты необходимо запускать на реальной PostgreSQL в CI/CD.
